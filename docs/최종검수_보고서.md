# 최종 검수 보고서 — 화룡점정

> 5가지 실제 시나리오 시뮬레이션 기반 정밀 검수

## 발견 이슈 5건

### F-1. 🔴 인라인 검증에 UX(섹션 3) 누락 — Phase 완료까지 UX 체크 공백

**현재 상태** (CL 섹션 0-2, line 127):
```
완료 후 — 인라인 검증 실행: 섹션 1(i18n 5단계) + 섹션 4-6(디자인 6단계)
```

**문제**: 인라인 검증에 i18n + 디자인만 있고 **UX(섹션 3)가 빠져 있습니다.**
3상태 구현, 엄지존, 피드백 같은 핵심 UX 항목이 Phase 완료까지 전혀 체크되지 않는 공백이 생깁니다.

i18n/디자인은 패턴 검색(문자열, Color값)으로 기계적 검증이 가능하지만,
UX는 설계 판단이 필요해서 Phase 완료까지 미루면 대규모 재작업 위험이 있습니다.

**수정안**: 인라인 검증을 3영역으로 확장
```
[인라인 검증] — 코드 생성 직후 매번 실행
  → 섹션 1 i18n 검증 (5단계) + 섹션 4-6 디자인 검증 (6단계)
  → 섹션 3 UX 핵심 체크 (3항목만 — 빠른 판단):
    ✓ 3상태(Loading/Empty/Error) 구현 여부
    ✓ 주요 CTA 엄지존(하단) 배치 여부
    ✓ 모든 액션에 상태 피드백 존재 여부
  → 리포트: "i18n 키 N개 / 디자인 위반 N건 / UX 이상 없음"
```

### F-2. 🟡 사전 확인 1번과 4번이 같은 섹션을 두 번 참조

**현재 상태** (CL 헤더, line 11·14):
```
1. 이 파일 "디자인 시스템" → AppColors 토큰 확인 (비어있으면 요청)
4. 이 파일 "디자인 시스템" → 다크Glass / 라이트파스텔 여부 확인
```

**문제**: 같은 "디자인 시스템" 섹션을 1번에서 열고, 3번(Phase) 확인하고, 다시 4번에서 열어야 합니다. AI가 1번에서 이미 그 섹션을 봤는데 4번에서 또 같은 곳을 참조하라고 하면 비효율입니다.

**수정안**: 1번에 합쳐서 5단계로 축소
```
1. 이 파일 "디자인 시스템" → AppColors 토큰 + 다크/라이트 스타일 확인 (비어있으면 요청)
2. 이 파일 "특이사항" → 터치 기준 / RBAC 역할 확인
3. 이 파일 "0-1 Phase" → 현재 Phase 범위 확인
4. DESIGN_MASTER PART 13·14 → 컴포넌트 + 페르소나 확인
5. SECURITY_MASTER PART 13 → 프로젝트 특화 보안 확인
```

### F-3. 🟡 리포트 형식이 4곳에서 미세하게 다름

| 위치 | 리포트 형식 |
|------|------------|
| CL 섹션 0 인라인 | `"i18n 키 N개 추가 / 디자인 위반 N건 수정"` |
| CL 섹션 1 | `"i18n 검증 완료 — 신규 키 N개 추가 (app_ko.arb / app_en.arb)"` |
| CL 섹션 4-6 | `"디자인 검증 완료 — 위반 N건 수정 (컬러N/터치N/그리드N)"` |
| DM ④ | `"디자인 검증 완료 — 위반 N건 수정 (컬러 N / 터치 N / 그리드 N / 기타 N)"` |

**문제**: AI가 "어떤 형식을 쓰지?" 4곳에서 형식이 다릅니다.

**수정안**: 인라인 검증 리포트를 유일한 출력 형식으로 선언하고, 섹션 1·4-6·DM④는 "내부 절차"로 격하
```
[인라인 검증 최종 리포트 — 이 형식이 유일한 출력]
"검증 완료 — i18n 키 N개 추가 / 디자인 위반 N건 (컬러N/터치N/그리드N) / UX 이상 없음"
※ 섹션 1·4-6의 개별 검증은 내부 절차. 리포트는 위 통합 형식 1회만 출력.
```

### F-4. 🟡 DM PART 8 헤더 "코딩 전 설계 검증에도 활용"이 인라인 검증과 혼란

**현재 상태** (DM PART 8, line 1407):
```
*코딩 전 설계 검증 / Phase 완료 후 QA / 최종 배포 전 전수 검사 모두 활용*
```

**문제**: CL에서는 PART 8 = "종합 검증 = Phase 완료 시"로 명확히 정의했는데, DM PART 8 자체 헤더에 "코딩 전 설계 검증에도 활용"이라고 적혀 있으면 AI가 매 화면에서도 PART 8을 돌릴 수 있습니다.

**수정안**: DM PART 8 헤더를 CL 정의에 맞게 수정
```
*⚠️ Phase 완료 시 전체 점검이 주 용도. 설계 단계에서 특정 항목 참고도 가능하나,
매 화면 인라인 검증(CLAUDE.md 섹션 0 참조)과 혼용 금지.*
```

### F-5. 🟢 SM "코딩 전 확인 순서"가 CL 헤더와 표현 충돌

**현재 상태** (SM, line 13):
```
코딩 전 확인 순서:
  1. CLAUDE.md "특이사항 > RBAC 역할" 확인
  2. CLAUDE.md "외부 서비스" 확인
  3. 이 파일 PART 13 확인
  4. 필요한 PART 상세 참조
```

**문제**: CL 헤더도 "코딩 전 필수 확인 순서"이고 SM도 "코딩 전 확인 순서"입니다. 같은 표현인데 내용이 다릅니다. SM의 것은 보안 작업 전용인데 표현이 구분되지 않습니다.

**수정안**: SM 표현을 "보안 작업 전 확인 순서"로 변경
```
보안 관련 코딩 전 확인 순서:
```
